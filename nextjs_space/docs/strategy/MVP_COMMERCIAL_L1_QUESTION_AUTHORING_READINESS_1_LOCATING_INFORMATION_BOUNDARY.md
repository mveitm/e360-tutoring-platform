## 7. Question-authoring readiness principles

Future question authoring, if separately authorized, should follow these principles:

* Stay tied to the reviewed draft text.
* Target `locating_information` only.
* Ask for one explicit detail at a time.
* Avoid inference beyond explicit information.
* Avoid purpose, viewpoint, broad main idea, and vocabulary interpretation.
* Use narrow wording so one answer is defensible.
* Require no external knowledge.
* Avoid memory overload.
* Require future alternatives and distractors to be reviewable for clear textual reasons.
* Require future feedback feasibility before accepting any item.

These principles define readiness boundaries only. They do not authorize item creation now.

## 8. Allowed future question target types, abstract only

A later authorized question-authoring phase may consider only abstract target types such as:

* explicit time;
* explicit place;
* explicit sequence or order;
* explicit object category;
* explicit role or person function in the fictional text;
* explicit condition;
* explicit action required by participants;
* explicit storage or location detail.

These are target categories only. They are not question wording, item plans, answer alternatives, distractors, answer keys, or feedback.

## 9. Blocked question target types

Blocked for this first locating-information boundary:

* inference beyond explicit information;
* purpose or viewpoint;
* vocabulary interpretation;
* broad main idea;
* opinion;
* external knowledge;
* multi-step reasoning;
* trick questions;
* questions requiring several details at once;
* questions that could have more than one defensible answer;
* questions requiring hidden context;
* questions that test memory of the whole passage rather than locating one explicit detail;
* questions that imply PAES-equivalence or official PAES item style.

Blocked target types require separate future authorization and review. They must not be created under a locating-information-only phase.

## 10. Answerability rules

Future locating-information questions should be allowed only if:

* the answer is directly supported by one explicit detail in the draft text;
* the question wording is narrow;
* one correct answer is defensible;
* alternatives and distractors can be wrong for clear textual reasons;
* no external knowledge is needed;
* the item does not overload memory;
* the item does not require several simultaneous details;
* feedback can explain the answer safely without long quotation;
* the item remains within `locating_information`.

If answerability is uncertain, the future item should be reworked or blocked before it becomes a question.

## 11. Future answer alternative boundary

This phase does not create answer alternatives.

Future answer alternatives, if separately authorized, should:

* include one defensible correct answer only;
* avoid introducing new facts not present in the text;
* avoid near-duplicate wording that could make multiple answers defensible;
* avoid outside knowledge;
* avoid misleading wording unrelated to the reading skill;
* remain compatible with future feedback.

Alternative review must happen before any question is accepted for a later content or implementation phase.

## 12. Future distractor boundary

This phase does not create distractors.

Future distractors, if separately authorized, should:

* be plausible but clearly wrong;
* be wrong for an identifiable textual reason;
* be based on nearby or similar explicit details only when this does not create accidental correctness;
* not require external knowledge;
* not introduce new facts;
* not create ambiguity;
* not be absurd or too obvious;
* be useful for future feedback.

Distractors that cannot be explained clearly should be rejected or reworked.

## 13. Future answer key boundary

This phase does not create answer keys.

Future answer keys, if separately authorized, should be accepted only when:

* the correct answer is supported by one explicit text detail;
* the answer key can cite or point to the relevant evidence safely;
* every distractor can be ruled out for a clear textual reason;
* no second answer is defensible under a reasonable reading;
* the key does not depend on external knowledge or hidden context.

Answer-key readiness is not product approval.

## 14. Future feedback dependency

This phase does not create feedback.

Future feedback must be able to:

* point to explicit evidence safely;
* explain why the correct answer is supported;
* explain why distractors fail;
* avoid long quotation where not needed;
* remain short and student-safe;
* avoid overdiagnosis;
* avoid AI tutor, score, mastery, theta, guaranteed improvement, or Sales-Ready claims.

If feedback cannot be written clearly and safely, the future question should not advance.

## 15. Recommended future question count range

If a later actual question-authoring phase is explicitly authorized, it should start with a very small set:

```text
2 to 4 locating-information questions maximum
```

This is not authorization to create those questions now.

The range is intentionally small because the first draft should test answerability, distractor control, and feedback feasibility before any broader L1 item set is considered.

## 20. Risks and mitigations

| Risk | Why it matters | Mitigation |
| --- | --- | --- |
| Boundary is mistaken for question authoring | Would bypass item review | State no questions, alternatives, distractors, keys, or feedback are created |
| Allowed target types become actual items | Would exceed documentation-only scope | Label targets as abstract categories only |
| Future items ask inference or purpose questions | Would exceed `locating_information` | Block those target types for this first phase |
| Distractors create ambiguity | Could make multiple answers defensible | Require future distractor review and clear textual reasons |
| Feedback is deferred until too late | Could create unexplainable questions | Require feedback feasibility before accepting items |
| Small draft is overused | Could produce repetitive or scanning-only work | Limit future first set to 2 to 4 items |
| Readiness language becomes product approval | Could weaken L1 and Sales-Ready gates | Keep student/product use and Sales-Ready fields as no/none |
