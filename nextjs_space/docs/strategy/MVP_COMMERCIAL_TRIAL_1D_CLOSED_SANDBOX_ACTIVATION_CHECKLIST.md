## 4. Purpose

The purpose of this checklist is to define the minimum readiness conditions before any future closed sandbox activation.

The checklist answers:

* Who exactly would be activated in a future phase?
* Is the participant class allowed?
* Is adult/minor status handled?
* Is the scope limited and understandable?
* Is the first learning step defined?
* Is route/content readiness available?
* Is there a support/failure path?
* Is copy approved and send authorized separately?
* Is StudentAccess/access-state handling understood?
* Are payment/minor questions still `NO-GO`?
* Is runtime access still unchanged?
* Are public Sales-Ready claims still blocked?
* What future phase would be required before actual activation?

## 5. What activation means and does not mean

For this roadmap block, "activation" means a future explicit decision to begin a closed, limited, supervised sandbox experience for a defined participant class.

Activation does not mean:

* Public launch.
* Public offer.
* Public Sales-Ready.
* Payment authorization.
* Trial activation.
* Runtime access change.
* Billing, subscription, or payment behavior.
* Automatic StudentAccess mutation.
* Complete PAES preparation.
* Market demand validation.
* Public pricing validation.

This phase defines activation readiness only. It does not authorize activation.

## 6. Participant readiness checklist

Before any future activation phase, the participant record must answer:

* Participant class identified.
* Participant is one of the allowed closed-lab classes for the future phase.
* Participant is not an external unknown participant unless a future gate explicitly authorizes that class.
* Adult/minor status is considered without storing unnecessary sensitive data.
* If minor status is relevant, guardian/consent/payment handling is resolved or activation remains blocked.
* Participant understands this is a closed laboratory.
* Participant understands this is not a public sale.
* Participant understands this is not complete PAES preparation.
* Participant objective is compatible with available scope.
* Participant does not require unlimited human support.
* Supervision owner is assigned.

Participant readiness is `BLOCKED` if the participant class is unclear, external/unknown, minor/payment policy is unresolved, or activation would create a public-offer signal.

## 7. Scope readiness checklist

Before activation, the sandbox scope must be:

* Closed.
* Non-public.
* Limited.
* Supervised.
* Understandable in plain language.
* Explicit about what is included.
* Explicit about what is not included.
* Clear about start and end/expiry candidate.
* Clear about pause/closure criteria.
* Free of complete PAES promise.
* Free of score promise.
* Free of public launch or Sales-Ready language.

Scope readiness is `FAIL` if the scope depends on hidden manual work to appear complete or if it implies M1/M2/L1 public readiness.

## 8. Route/content readiness checklist

Before activation, route/content readiness must show:

* A first learning step is defined.
* The student can understand what to do first.
* The initial route or activity is compatible with the participant objective.
* The route does not claim complete PAES coverage.
* The route can produce observable evidence.
* The route does not require a human to manually create the experience outside the app as normal operation.
* If actual app operation is needed to prove readiness, that must be handled in a future explicit technical/readiness phase.

Current 1D decision: route/content readiness is not verified live in this phase. Activation remains future.

## 9. Student experience readiness checklist

Before activation, the student-facing experience must be ready enough that:

* The student knows the first step.
* The student sees a limited, calm, non-public framing.
* The student is not exposed to internal admin state.
* The student is not shown `StudentAccess` as a commercial promise.
* The student is not promised complete PAES preparation.
* The student is not promised a score result.
* The student is not promised advanced AI tutoring.
* The student is not promised automatic continuity.
* Any visible copy is approved in a future send/use phase.

1D does not change student UI and does not approve any visible runtime behavior.

## 10. Copy/send readiness checklist

Before any future activation:

* Invitation/send phase is explicitly opened and passed.
* Recipient class is documented.
* Send channel is documented.
* COPY-2 internal copy is rechecked against current scope.
* Copy still says closed laboratory.
* Copy avoids public launch, public sale, complete PAES preparation, score promise, payment, price, subscription, billing, and automatic access.
* Copy does not expose `StudentAccess`.
* Copy does not imply invitation equals activation.
* Copy does not imply activation equals payment or subscription.
* If minors are involved, Pro/high-level review decides the communication path.

COPY-2 is an internal artifact only. It is not sent by 1D.

## 11. Support/failure readiness checklist

Before activation, support/failure readiness must define:

* Human supervisor.
* Technical operator, if needed.
* Support scope.
* What support can do.
* What support must not do.
* How blockers are recorded.
* How the sandbox pauses.
* How the sandbox closes.
* What happens if the student cannot start.
* What happens if route/content is insufficient.
* What happens if evidence is not visible.
* What happens if support burden becomes high.
* What happens if copy causes misunderstanding.

Support must not become unlimited manual tutoring or the mechanism that makes an incomplete product appear complete.

## 12. StudentAccess/access-state readiness checklist

Before activation, access-state readiness must answer:

* Is current StudentAccess state known through an authorized future check?
* Does activation require any StudentAccess transition?
* If yes, has that transition been authorized by a future phase?
* Is runtime access behavior aligned with StudentAccess state?
* Is missing-row policy safe?
* Is repair/autocreate out of scope unless separately authorized?
* Is StudentAccess kept distinct from payment, invitation, runtime access, and public promise?

Current state:

* StudentAccess default remains non-permission.
* Admin `reaffirm_no_access` is available and smoke-verified.
* `reaffirm_no_access` does not activate access, trial, or billing.
* No StudentAccess mutation is authorized in 1D.

## 13. Payment/minor readiness boundary

Payment remains `NO-GO`.

Activation with payment is blocked until Pro/high-level review resolves:

* Whether payment can be accepted at all.
* Whether minors can participate in any payment context.
* Payer identity.
* Guardian/consent path.
* Price.
* Refund/pause policy.
* Payment confirmation path.
* Billing/payment source of truth.
* Support and failure handling when money is involved.

This phase does not approve payment copy and does not authorize real payment.

## 14. Data/privacy readiness checklist

Before activation, data/privacy boundaries must be clear:

* Use minimum necessary personal data.
* Avoid unnecessary minor details in general docs.
* Use redacted/partial identifiers in reports.
* Do not expose emails unless needed in a controlled internal context.
* Do not include payment details in general docs.
* Do not expose cookies, tokens, headers, session payloads, raw environment values, credentials, or secrets.
* Keep evidence educational and operational.
* Separate private participant notes from public/product claims.

1D performs no DB read and accesses no live participant data.

## 15. Technical readiness boundary

This checklist does not operate the app or verify technical readiness live.

Technical readiness remains future if activation would require:

* App operation.
* Browser smoke.
* API calls.
* DB reads.
* DB mutations.
* Runtime access changes.
* `/now` behavior changes.
* Student UI changes.
* Admin UI changes.
* Endpoint changes.
* Auth changes.
* StudentAccess mutation.
* Trial activation.
* Billing/subscription/payment behavior.
* Deploy.

Any such requirement must be handled by a separate explicit phase.

## 16. PASS criteria

Activation-readiness documentation is `PASS` if:

* Participant class criteria are defined.
* Scope readiness criteria are defined.
* Route/content readiness criteria are defined.
* Student experience readiness criteria are defined.
* Copy/send prerequisites are defined.
* Support/failure readiness is defined.
* StudentAccess/access-state prerequisites are defined without changing them.
* Payment/minor boundaries remain `NO-GO`.
* Runtime access remains unchanged.
* Public Sales-Ready remains blocked.
* Future phases required before real activation are named.

1D meets this only as documentation readiness.

## 17. FAIL criteria

Activation-readiness fails if:

* The checklist is treated as activation authority.
* Any invitation is sent.
* Any participant is contacted.
* Any minor is involved.
* Any payment copy is approved.
* Any payment is requested or authorized.
* Any sandbox, trial, runtime access, billing, subscription, payment, `/now`, student UI, admin UI, StudentAccess, DB, code, staging, production, or deploy behavior changes.
* The scope implies public Sales-Ready.
* The scope claims complete PAES preparation.
* The scope treats owner/operator or known-user planning as market validation.

## 18. BLOCKED criteria

Future activation is blocked if:

* Participant class is unclear.
* Recipient has not been invited through an explicit future send phase.
* Copy has not been approved for real send in the future phase.
* Adult/minor handling is unresolved.
* Payment is required.
* Minor/payment/consent policy is unresolved.
* First learning step is missing.
* Route/content readiness is unverified.
* App cannot guide the first step.
* Support/failure path is missing.
* StudentAccess/access-state handling is inconsistent.
* Runtime access changes are required but not authorized.
* Billing/subscription/payment behavior is required.
* `/now` or student UI changes are required.
* Public offer or Sales-Ready decision is required.
