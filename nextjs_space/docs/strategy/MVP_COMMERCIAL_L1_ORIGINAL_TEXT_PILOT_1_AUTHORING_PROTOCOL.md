# MVP-COMMERCIAL-L1-ORIGINAL-TEXT-PILOT-1 - Original-Text Pilot Authoring Protocol

## 1. Phase

* Phase: `MVP-COMMERCIAL-L1-ORIGINAL-TEXT-PILOT-1`.
* Type: documentation-only / L1 original-text authoring protocol / pre-content governance.
* Product horizon: roadmap change-control / L1 original-text pilot preparation / Sales-Ready support.
* This is not `MVP-Beta-Sales-Ready` approval.
* M2 remains not ready.

## 2. Relationship to L1-TEXT-RIGHTS-1 and L1-TEXT-QUALITY-1

`MVP-COMMERCIAL-L1-TEXT-RIGHTS-1` defined the conservative source/text rights and editorial policy boundary for future L1 content. It established that original Bexauri-created texts are the preferred candidate source class for early L1 pilot content, while still requiring editorial, difficulty, bias/sensitivity, ambiguity, answerability, feedback feasibility, and metadata review.

`MVP-COMMERCIAL-L1-TEXT-QUALITY-1` converted that policy into a future reviewer checklist for text quality, source/rights metadata, cognitive load, difficulty, ambiguity, answerability, question quality, distractor quality, feedback feasibility, sensitivity, accessibility, review states, and PASS / FAIL / BLOCKED criteria.

This document defines how future original Bexauri-created L1 pilot texts could be authored, reviewed, versioned, and approved for later content phases. It does not create or approve any actual text.

## 3. Nature of document

This is an authoring protocol, not text creation.

It is:

* documentation-only;
* pre-content governance;
* limited to original Bexauri-created text concepts;
* intended to prepare future content-authoring phases without starting them.

It is not:

* text selection;
* text creation;
* question creation;
* distractor creation;
* feedback creation;
* StudyLoad design;
* content registry authority;
* official PAES/DEMRE text or question approval;
* copyrighted text approval;
* L1 readiness approval;
* Sales-Ready approval.

## 4. Purpose

The purpose is to define a safe protocol for future original Bexauri-created L1 pilot text work before any text, question, distractor, StudyLoad, registry, feedback, dashboard, or implementation phase begins.

This phase defines:

* conceptual author and reviewer roles;
* allowed early text goals;
* pre-draft and post-draft metadata needs;
* drafting constraints for original texts;
* reading-skill targeting rules;
* review gates before a text can move to a later content-authoring phase;
* question, distractor, and feedback boundaries for later phases;
* PASS / FAIL / BLOCKED criteria;
* risks, reserved questions, and future phase options.

## 5. Protocol principles

Future original-text pilot work should follow these principles:

* Prefer original Bexauri-created texts for early L1 pilot candidates because rights custody is clearest.
* Keep every original text short or medium length for early pilot work.
* Give each text one clear reading purpose.
* Give each text one primary reading-skill target.
* Keep vocabulary, sentence complexity, context, and cognitive load controlled.
* Avoid hidden external-knowledge dependencies.
* Use low-sensitivity topics by default.
* Ensure textual evidence would be available for future questions.
* Do not copy official PAES/DEMRE passages, questions, or item wording.
* Do not copy or imitate a specific copyrighted text.
* Do not claim that an original text is official PAES-like, PAES-equivalent, or endorsed.
* Treat PASS as readiness only for a later content-authoring or review phase, not product use.

## 6. Original-text pilot scope boundary

This protocol applies only to future original Bexauri-created text concepts for an early L1 pilot.

Allowed future concept scope:

* original short or medium texts;
* low-sensitivity non-literary or simple literary-style texts only if approved by future editorial direction;
* text-bound reading tasks;
* candidate reading-skill targeting based on prior accepted docs;
* metadata and review-state preparation.

Not allowed in this phase:

* actual text drafting;
* actual passages;
* actual questions;
* actual distractors;
* actual feedback;
* StudyLoads;
* registry records;
* app implementation;
* student-facing use;
* public readiness or Sales-Ready claims.

## 7. Candidate authoring roles, conceptual only

Future phases may define roles such as:

* `text_author`: drafts original Bexauri text under approved constraints.
* `reading_skill_owner`: confirms the intended reading-skill target is valid for the future phase.
* `metadata_custodian`: records source class, rights status, version, reviewer states, and approval boundaries.
* `editorial_coordinator`: routes the asset through future review states.

These roles are conceptual only. This phase does not assign people, create workflow tooling, or authorize authoring.

## 8. Candidate reviewer roles, conceptual only

Future phases may define roles such as:

* `rights_reviewer`: confirms the text is original Bexauri-created and not copied or adapted from a protected source.
* `editorial_reviewer`: reviews clarity, length, vocabulary, context, tone, and student suitability.
* `reading_reviewer`: checks reading purpose, reading-skill fit, evidence availability, and answerability potential.
* `sensitivity_reviewer`: checks bias, age/context fit, and sensitive-topic risk.
* `feedback_feasibility_reviewer`: checks whether future feedback could safely point to evidence and explain reasoning.
* `final_phase_reviewer`: decides whether the concept may move to a later content-authoring or review phase.

These roles are conceptual only. Legal-sensitive issues remain Pro/high-level/legal/editorial reserved.

## 9. Pre-draft metadata requirements

Before any future original text is drafted, the future phase should define metadata for:

* proposed text ID;
* proposed internal title;
* source class: original Bexauri-created text;
* author or drafting owner;
* intended L1 reading-skill target;
* intended reading purpose;
* intended text format;
* target length band;
* target difficulty band;
* target vocabulary constraints;
* target sentence-complexity constraints;
* sensitivity risk expectation;
* intended use: pilot candidate only;
* prohibited source inputs;
* reviewer states required before next phase;
* version seed, such as `draft-0` or equivalent future standard.

If this metadata is missing, future drafting should not start.

## 10. Original-text drafting constraints

Future original-text drafting should be constrained as follows:

* short or medium length only for early pilot;
* one clear reading purpose;
* one primary reading-skill target;
* controlled vocabulary;
* controlled sentence complexity;
* clear context;
* no hidden external-knowledge dependency;
* low sensitivity by default;
* textual evidence available for future questions;
* no copied official/copyrighted passage;
* no imitation of a specific copyrighted text;
* no public claim that the text is official PAES-like or PAES-equivalent;
* no reliance on a private cultural reference required to answer;
* no dense multi-topic structure unless a future reviewer approves it;
* no ambiguity that would prevent one defensible multiple-choice answer in later phases.

## 11. Reading-skill targeting protocol

Future original-text concepts may target only candidate skill areas already supported by prior docs:

* locating information;
* interpreting meaning and relationships;
* evaluating text, context, form, purpose, and position;
* text-bound contextual vocabulary only if justified;
* relationship between ideas and structure only if supported.

Protocol:

1. Choose one primary reading-skill target before drafting.
2. State why the text can support that skill.
3. Identify what textual evidence would support future questions.
4. Avoid inventing detailed official subskills from memory.
5. Keep integrated or mixed-skill use for later phases unless a future taxonomy boundary allows it.

This does not create a final L1 taxonomy.

## 12. Text length / load / difficulty protocol

Early original-text candidates should:

* remain short or medium length;
* avoid excessive information density;
* use vocabulary appropriate to the target student;
* avoid sentence structures that distract from the intended reading skill;
* avoid too many speakers, viewpoints, time shifts, or abstract relationships;
* keep inference burden controlled;
* support future question answerability without requiring rereading large sections;
* be feasible for feedback without long quotation blocks.

Difficulty must be intentional, not accidental. A text is too difficult if its complexity comes from unclear context, vague wording, unsupported references, or overloaded structure.

## 13. Sensitivity / bias / context protocol

Future original-text concepts should default to low-sensitivity topics.

Review should check:

* age/context appropriateness;
* avoidable political, religious, traumatic, violent, sexual, health, legal, or identity-sensitive content;
* stereotype or bias risk;
* cultural/context clarity;
* whether background knowledge creates an unfair barrier;
* whether the topic is necessary for the reading purpose;
* whether Pro/high-level/editorial approval is required.

Sensitive or controversial text concepts should be `BLOCKED` by default for early L1 until separately approved.

## 14. Ambiguity / answerability protocol

Before a future original text can move to question-authoring eligibility, reviewers should confirm:

* the reading purpose is clear;
* textual evidence exists for future questions;
* likely future multiple-choice questions could have one defensible correct answer;
* reasonable alternate interpretations have been considered;
* the text does not rely on unstated external knowledge;
* vocabulary-in-context tasks remain text-bound;
* purpose, viewpoint, and structure tasks can be answered from the text;
* feedback could explain the answer in student-safe language.

If the text cannot support answerable questions, it should be `FAIL` or `BLOCKED` depending on whether rework can solve the issue.

## 15. Question/distractor later-phase boundary

This phase does not create questions or distractors.

This protocol may only define eligibility for a later question-authoring phase.

Future eligibility should require:

* completed source/rights metadata;
* original-text status reviewed;
* reading-skill target recorded;
* text quality checklist passed for next phase;
* evidence locations or evidence references identified conceptually;
* ambiguity and answerability risks reviewed;
* feedback feasibility reviewed at concept level.

Future question phases must separately define question wording, correct answer, distractors, rationales, and review workflow.

## 16. Feedback-feasibility later-phase boundary

This phase does not create feedback.

Future original-text concepts should be reviewed for whether later feedback could:

* point to evidence safely;
* explain why the correct answer is supported;
* explain why distractors fail;
* avoid long quotations;
* name a likely reading behavior or error without overdiagnosis;
* remain calm, clear, and student-safe;
* stay compatible with source/rights metadata;
* avoid unsupported AI, score, mastery, or guaranteed-improvement claims.

If feedback cannot be made feasible, future question authoring should not proceed.

## 17. Versioning and change-control protocol

Future original-text concepts should use explicit versioning before and after drafting.

Minimum future versioning rules:

* assign a stable text ID before drafting;
* record each version with date, author/editor, and change reason;
* preserve whether the text is pre-draft, draft, review, rework, blocked, or passed for next phase;
* record whether the reading-skill target changed;
* record whether difficulty, sensitivity, or answerability changed;
* record whether a change requires re-review;
* never overwrite review history silently;
* do not publish, implement, or register a text because it has a version.

Versioning is custody, not approval.

## 18. PASS criteria

For this protocol, future concept-level `PASS` means only:

* the future original-text concept has enough metadata to move to a later content-authoring or review phase;
* the source class is original Bexauri-created and rights custody is clear at concept level;
* the reading purpose and primary skill target are clear;
* length, load, difficulty, sensitivity, ambiguity, and answerability constraints appear manageable;
* future question and feedback feasibility appear plausible;
* no unresolved source, rights, editorial, sensitivity, metadata, or reviewer-authority blocker remains.

PASS does not approve product use, student use, L1 readiness, official PAES/DEMRE use, copyrighted text use, or Sales-Ready.

## 19. FAIL criteria

Future original-text concepts should be marked `FAIL` if:

* reading purpose is unclear;
* primary reading-skill target is unclear;
* the concept is too broad, too dense, or too vague for early pilot;
* context or vocabulary would create unfair difficulty;
* the future text would likely require external knowledge;
* answerable future questions are unlikely;
* feedback feasibility is weak;
* the concept reduces L1 to generic quiz behavior;
* rework is needed before any later phase.

FAIL means unsuitable as conceived or needing rework. It does not authorize replacement content.

## 20. BLOCKED criteria

Future original-text concepts should be marked `BLOCKED` if:

* the source is not actually original Bexauri-created;
* the concept depends on official PAES/DEMRE text or question wording;
* the concept depends on copyrighted material or imitation of a specific copyrighted work;
* sensitivity, bias, or age/context risk requires Pro/high-level/editorial direction;
* metadata is missing;
* reviewer authority is unresolved;
* source or rights conflict appears;
* the concept would imply product use, student use, L1 readiness, or Sales-Ready before later phases.

BLOCKED means do not draft, author questions, implement, register, or use until the blocker is resolved in a separate phase.

## 21. Risks and mitigations

| Risk | Why it matters | Mitigation |
| --- | --- | --- |
| Protocol is mistaken for actual text approval | Could bypass content, editorial, and implementation gates | State this is authoring protocol only and PASS is next-phase readiness only |
| Original text is treated as automatically safe | Original writing can still be ambiguous, biased, too hard, or pedagogically weak | Require editorial, difficulty, sensitivity, answerability, and feedback feasibility review |
| Author copies or imitates a protected source | Creates rights and brand risk | Require original-source metadata and no imitation of specific copyrighted texts |
| L1 becomes generic quiz content | Breaks the reading-specific identity required by prior docs | Require one reading purpose, one primary skill target, and textual evidence |
| Feedback feasibility is deferred too long | Later questions may be unexplainable | Require feedback feasibility before question-authoring eligibility |
| PASS becomes product use | Creates readiness drift | Define PASS as readiness only for later content-authoring/review phase |
