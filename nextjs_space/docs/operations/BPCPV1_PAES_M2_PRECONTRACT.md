# BPCPv1-PAES-M2 - Precontrato minimo antes de derivacion

## 1. Executive verdict

```text
READY_FOR_DERIVATION_WITH_CONDITIONS
```

Bexauri esta en condiciones de iniciar la derivacion documental de BPCPv1-PAES-M2, siempre que esta derivacion no se confunda con implementacion tecnica, piloto real, cobertura completa, Program DB row, enrollment, smoke humano ni agentic.

Este precontrato define minimos obligatorios antes de ejecutar tecnicamente PAES_M2 como tutoria Bexauri.

La derivacion puede avanzar porque ya existen BPCPI como ideal rector, BPTT como gramatica transversal, BPTT Roadmap / Living Microroadmap Contract como contrato de ruta pedagogica, Blueprint M2 Integrado como contrato tactico-pedagogico especifico, M2-C01 como primer artefacto authored implementado y static-reviewed, y estandares PAES de capsulas, feedback, visual stimuli y FigureSpec.

BPCPv1-PAES-M2 debe nacer con restricciones fuertes: no declarar cobertura completa, score, mastery, readiness real ni adaptive AI; no crear Program sin contrato; no crear enrollment huerfano; no usar M2-C01 como sustituto de tutoria M2 completa.

## 2. Purpose of this precontract

Este precontrato responde:

```text
Que debe estar decidido antes de derivar o ejecutar BPCPv1-PAES-M2?
```

Su funcion es impedir que PAES_M2 nazca como extension tecnica de M1, fila DB, coleccion de ejercicios, ruta de smoke, promesa comercial o adaptacion inteligente sin base.

Este documento no es BPCPv1 global, implementacion, roadmap tecnico, schema, endpoint, UI, fixture, seed script, banco final de items ni contrato de cobertura completa. Si es frontera pedagogica de seguridad, lista de minimos antes de Program, proteccion contra activacion tecnica prematura y puente entre blueprints y ejecucion.

## 3. Boundary: BPCPv1 global vs BPCPv1-PAES-M2

BPCPv1 es la futura capa pedagogica v1 global de Bexauri. BPCPv1-PAES-M2 es una derivacion especifica para PAES Matematicas M2.

```text
BPCPv1 != BPCPv1-PAES-M2
```

BPCPv1-PAES-M2 no gobierna PAES_M1, PAES_L1, tutorias skill-oriented futuras, toda la arquitectura pedagogica global, pagos, branding, auth, UI concreta, schema ni motor adaptativo.

Si gobierna definicion pedagogica de PAES_M2, capsulas M2, evidencia M2, feedback M2, autorreporte M2, revision M2, continuidad M2, limites de inferencia M2 y condiciones pedagogicas antes de Program PAES_M2.

## 4. Required inheritance from BPCPI

BPCPv1-PAES-M2 hereda:

- actividad no equivale a aprendizaje;
- toda practica debe tener proposito;
- toda practica debe producir evidencia;
- toda evidencia tiene limites;
- todo feedback debe orientar accion;
- toda continuidad debe justificarse;
- el estudiante debe saber que hace y que sigue;
- el error es informacion;
- la automatizacion debe ser supervisable;
- el progreso debe ser honesto.

## 5. Required inheritance from BPTT

BPCPv1-PAES-M2 debe expresar la gramatica transversal:

```text
proposito
dominio/habilidad
progresion
recursos
capsulas
evidencia
feedback
autorreporte
continuidad
progreso
revision
```

Debe mantener PAES_M2 como tutoria test-oriented, recordando que la prueba es contexto y no esencia. El objetivo es desarrollar destrezas matematicas, no solo maximizar aciertos.

## 6. Required inheritance from roadmap/microroadmap contract

PAES_M2 debe distinguir secuencia simple, roadmap maestro M2 y microroadmap vivo del estudiante.

En etapa inicial, M2-C01 pertenece a una secuencia simple. No declara roadmap maestro ni microroadmap vivo.

Antes de declarar roadmap maestro M2 se requiere mapa de familias/ejes, nodos, prerrequisitos, niveles, cobertura, tipos de item, errores frecuentes, criterios de avance/refuerzo/revision y fuentes oficiales o criterios documentados.

Antes de declarar microroadmap vivo M2 se requiere evidencia acumulada por estudiante.

## 7. Required M2-specific decisions before implementation

Antes de ejecutar tecnicamente PAES_M2 se debe decidir proposito M2, limites MVP cerrado, familias iniciales, tipos de capsulas, estandar de feedback, visual/math dependencies, relacion con M2-C01, condiciones para Program, condiciones para enrollment y condiciones para smoke.

## 8. What can be canonized now

Puede canonizarse PAES_M2 como tutoria test-oriented Bexauri, M2-C01 como primer artefacto authored, maximo 4 ejercicios por capsula M2, necesidad de authoredFeedback brief/complete, necesidad de estandares visuales/LaTeX/figuras, prohibicion de agentic productivo y regla de que Program creation debe seguir contrato pedagogico.

## 9. What must be corrected or added before Sales-Ready

Antes de Sales-Ready se debe agregar/corregir cobertura authored M2 suficiente, roadmap maestro M2, mas capsulas M2, QA de feedback, QA de visual stimuli, ruta de acceso/enrollment estable, smoke humano, regresion del set M2, criterios de continuidad y limites de claims.

## 10. What must be blocked

Bloquear Program PAES_M2 en prod/staging, enrollment sin Program, Program sin contrato pedagogico, smoke que oculte ausencia de ruta, M2 como cobertura completa, M2 Sales-Ready, agentic, score/readiness/mastery, visual stimuli no auditados y feedback key-check-only.

## 11. Relationship to M2-C01

M2-C01 puede validar circuito inicial, iniciar smoke cuando exista access y demostrar authored content inicial. No define toda la tutoria ni autoriza Program sin contrato, cobertura M2 completa, roadmap maestro, agentic ni Sales-Ready.

## 12. Conditions for creating Program PAES_M2

Condiciones:

- blueprint M2 integrado creado;
- precontrato M2 creado;
- governing document M2 creado;
- local/dev only;
- no prod/staging;
- no schema;
- no payment/subscription;
- no enrollment en la misma fase;
- Program metadata espeja patron PAES_M1 donde aplique;
- no cambios M2-C01;
- no agentic.

## 13. Conditions for enrollment/access

Despues de Program, StudentAccess o enrollment debe estar separado, local/dev only, sin schema, sin payment/subscription real, sin route-order/continuity salvo fase explicita; debe preparar acceso a ST M2 y permitir smoke M2-C01.

## 14. Conditions for future authored roadmap M2

Antes de ampliar M2 se deben definir familias/ejes prioritarios, establecer coverage targets, revisar dependencias visuales/math, planificar capsulas adicionales, validar feedback y no abrir agentic hasta banco authored suficiente.
