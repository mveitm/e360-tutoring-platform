# Bexauri — Estrategia económica, técnica y pedagógica para generación de cápsulas completas con API externa y mejora continua

**Phase:** `MVP-SALES-PILOT-PEDAGOGY-M1-AI-CAPSULE-GENERATION-CONTINUOUS-IMPROVEMENT-STRATEGY-1`
**Nature:** Pedagogical/product/technical strategy documentation.
**Scope:** Documentation only. No implementation.

## 1. Executive verdict

```text
API_COMPLETE_CAPSULE_GENERATION_AND_CONTINUOUS_IMPROVEMENT_STRATEGY_READY
```

Bexauri should not build a large manual authored capsule bank before validating market demand and real student use.

The accepted strategy is to combine controlled authored early capsules with future external API generation of complete pedagogical capsules, automatic Bexauri validation, non-blocking assignment when validation passes, supervised quality review, a living reusable capsule bank, and continuous improvement through evidence of use.

This document does not implement API calls, does not create generated capsules, does not modify registry, does not change continuity, and does not claim adaptive AI or Sales-Ready status.

## 2. Strategic decision

Bexauri should use:

- authored controlled initial capsules;
- generation of complete pedagogical capsules by external API after the controlled early window;
- API output that includes items, answer keys, reasoning, feedback, full feedback, step-by-step help, errors, metadata, inference limits, and continuity suggestion;
- automatic Bexauri validation before assignment;
- assignment without ordinary human blocking if validation passes;
- human supervision focused on performance and feedback quality;
- a living reusable capsule bank;
- continuous improvement through evidence of use;
- cost control through reuse and generation only when there is a pedagogical gap.

This is a strategy for the future pedagogical layer. It is not an API integration spec and not implementation authorization.

## 3. Why this strategy is needed now

SOURCE-AUDIT-1 showed that current M1 content has gaps against the 2-6 calibration plan and lacks authored feedback, full feedback, item rationale, item metadata, and step-by-step help in inspected M1 registry entries.

Manually authoring a complete bank before validating the product would be expensive, slow, and brittle. However, assigning raw exercises generated on demand would repeat the same pedagogical problem in another form: activity without enough feedback, review, metadata, and continuity.

The strategic answer is not "generate exercises." The answer is to generate complete capsules under Bexauri governance, validate them automatically, reuse what works, and continuously improve the bank from evidence.

## 5. Unit generated by API

The unit generated by API is a complete pedagogical capsule, not an exercise.

A complete capsule must include, as one structured output:

- capsule identity and pedagogical purpose;
- target tutoring/program and roadmap node;
- items or tasks;
- answer keys;
- expected reasoning;
- item-level feedback;
- full feedback;
- step-by-step help;
- expected errors and distractor rationale;
- metadata;
- difficulty and prerequisite assumptions;
- evidence expected;
- inference limits;
- prohibited claims;
- continuity suggestion.

An exercise without these components is not enough for BPCPv1-PAES-M1.

## 6. Why API must generate complete capsules, not exercises only

BPCPv1-PAES-M1 requires each capsule to transform student activity into evidence, feedback, review, and cautious continuity. A bare exercise only produces a response. It does not by itself explain what the student should review, what error may have appeared, what the next action means, or what Bexauri is not allowed to infer.

If the API only generates exercises, Bexauri would still need manual work for:

- answer validation;
- reasoning validation;
- feedback design;
- step-by-step help;
- error taxonomy;
- metadata;
- continuity language;
- inference limits.

Therefore the API output must be capsule-complete from the start, even if Bexauri later validates, edits, versions, or rejects it.

## 7. Recommended architecture

Recommended conceptual architecture:

```text
Pedagogical intent
-> Bexauri generation brief
-> external API complete capsule generation
-> Bexauri automatic validation
-> candidate capsule record
-> assignment if validation passes and no better reusable capsule exists
-> student use
-> evidence collection
-> supervision and quality review
-> living capsule bank update
```

Key boundaries:

- Bexauri owns the roadmap, validation, assignment policy, bank versioning, and claims.
- The external API drafts complete capsule candidates.
- Human supervision audits quality, anomalies, and improvement decisions.
- No ordinary student continuity should depend on Mauricio manually prompting a personal ChatGPT account.

## 8. When real generation should activate

Real generation should activate only when there is a pedagogical gap that cannot be satisfied by an approved reusable capsule.

Examples:

- missing capsule concept in a required early window;
- required node/focus has no validated capsule in the bank;
- student evidence suggests a needed variation and no approved variation exists;
- a banked capsule is unsuitable because of difficulty, metadata, or known quality issues;
- a planned roadmap segment needs a source packet before ficha or implementation.

Generation should not activate just because a student completed an activity or because novelty is desirable. Reuse must be checked first.

## 9. Shadow mode recommendation

Before student-facing assignment, Bexauri should run future generation in shadow mode.

Shadow mode means:

- API generates candidate capsules from Bexauri briefs;
- Bexauri validates them automatically;
- candidates are logged and reviewed;
- they are not assigned to students automatically;
- humans compare generated candidates against BPCPv1 expectations and known gaps;
- validation failure patterns are used to improve the future contract.

Shadow mode should precede live generation because it exposes structural, mathematical, metadata, feedback, and cost risks without affecting students.

## 10. What Bexauri generates before API call

Before calling an external API, Bexauri should generate or assemble a strict generation brief:

- tutoring/program: for example `PAES_M1`;
- roadmap node and axis;
- capsule role: practice, bridge, validation, reinforcement, coverage, or review;
- intended position in the sequence or microroadmap hypothesis;
- prerequisite assumptions;
- difficulty band;
- item count and allowed item types;
- expected evidence;
- expected error families;
- feedback requirements;
- step-by-step requirements;
- prohibited claims;
- language/tone constraints;
- source constraints;
- metadata schema target;
- output format;
- validation checklist.

The external API should not decide the pedagogy from scratch.

## 11. What API must return

The API must return a structured complete capsule, including at least:

- capsule title;
- stable draft key or candidate key;
- program;
- axis/family/node metadata;
- purpose and non-purpose;
- item list;
- answer keys;
- reasoning for correct answers;
- distractor rationale or expected error interpretation;
- brief feedback per item;
- full feedback per item;
- step-by-step help per item;
- capsule-level feedback;
- self-report interpretation guidance;
- evidence expected;
- continuity suggestion;
- inference limits;
- prohibited claims;
- difficulty estimate;
- prerequisite assumptions;
- validation self-check;
- version metadata.

Each generated capsule must include feedback and step-by-step help in the same structured output. Missing feedback or missing step-by-step means the generated output is incomplete.

## 12. What Bexauri must validate automatically

Bexauri must validate automatically before assignment:

- output structure;
- required fields present;
- item count and item type;
- answer keys;
- reasoning;
- mathematical consistency;
- feedback;
- full feedback;
- step-by-step help;
- error and distractor logic;
- metadata;
- difficulty band;
- prerequisite assumptions;
- inference limits;
- prohibited claims;
- no PAES score claim;
- no theta claim;
- no mastery claim;
- no adaptive AI claim;
- no diagnostic overclaim;
- no unsupported readiness claim;
- no unsafe or irrelevant content;
- compatibility with BPCPv1-PAES-M1 and BPTT constraints.

Validation must be a Bexauri responsibility. The API's own confidence is not enough.

## 13. Assignment without ordinary human blocking

Bexauri may assign a generated capsule without ordinary human approval only when automatic validation passes and assignment policy allows it.

This preserves the non-blocking operator principle:

```text
review != gate
supervision != bottleneck
operator action != prerequisite for normal student continuity
```

This does not mean all generated capsules are automatically accepted forever. It means routine assignment can proceed when the system has a validated capsule candidate and no exceptional risk is present.

Exceptional cases may still require human review:

- validation failure;
- mathematical ambiguity;
- repeated poor performance;
- anomaly in feedback quality;
- high-impact route decision;
- new capsule family;
- suspicious generation pattern;
- sensitive student-facing language risk.

## 14. Human supervision role

Human supervision should focus on:

- performance supervision;
- feedback quality;
- step-by-step quality;
- student confusion patterns;
- anomaly detection;
- repeated validation failures;
- bank quality decisions;
- version promotion or retirement;
- roadmap/focus correction;
- cost and reuse review.

Humans should not be the ordinary gate for every student next action. They improve the system asynchronously and intervene when evidence, quality, or risk justifies it.

## 15. Living capsule bank

Capsules approved by validation and supported by use evidence feed a living reusable bank.

The bank should track:

- capsule identity;
- version;
- source/generation origin;
- validation status;
- assigned tutoring/program;
- node and difficulty metadata;
- use count;
- performance patterns;
- feedback usefulness signals;
- known issues;
- revision history;
- retirement or promotion status.

The living bank is central to cost control and quality growth. Bexauri should prefer reusing an approved capsule over generating a new one when the pedagogical need matches.

## 16. Continuous improvement strategy

Continuous improvement means Bexauri learns from use evidence to improve capsule quality, bank coverage, feedback, step-by-step help, metadata, and continuity suggestions.

It should be:

- governed;
- supervised;
- versioned;
- evidence-aware;
- auditable;
- conservative about claims.

Continuous improvement does not mean autonomous adaptive AI. It does not mean the system can claim mastery, theta, PAES score, exact personalization, or complete readiness.

Improvement signals may include:

- repeated wrong distractor patterns;
- mismatch between feedback and student recovery;
- high "Me costo" despite high correctness;
- low correctness with high confidence;
- frequent step-by-step use;
- repeated abandonment;
- unexpected difficulty;
- high reuse with stable performance;
- human quality review.

## 17. Cost strategy

API usage has cost and must be controlled.

Cost controls:

- reuse approved banked capsules first;
- generate only when there is a pedagogical gap;
- use shadow mode before live broad use;
- validate structure before deeper review;
- cache generated candidates;
- version rather than regenerate when small edits suffice;
- retire low-quality capsules;
- track generation cost by node/family/use case;
- avoid per-student generation when a reusable capsule fits.

Reuse of approved capsules is central to keeping cost from scaling linearly with students.

Bexauri must not depend on Mauricio's personal ChatGPT account. Future generation must use controlled product infrastructure, governed API access, cost tracking, and audit logs.

## 18. What is generated once vs per student

Generated once or rarely:

- standard capsule candidates for roadmap nodes;
- common feedback/step-by-step variants;
- source packets for missing concepts;
- reusable bridge capsules;
- reusable validation capsules;
- banked variations by difficulty and focus.

Generated per student only when justified:

- a specific variation required by evidence and not present in the bank;
- a bridge needed because of a rare evidence combination;
- an internal decision-support candidate, not a public claim;
- an exceptional remediation/validation need after reuse check fails.

Default rule:

```text
generate for the bank when possible;
generate per student only when the bank cannot satisfy a real pedagogical gap.
```

## 19. Risks and mitigations

| Risk | Mitigation |
|---|---|
| API generates plausible but wrong math | Validate answer keys, reasoning, and consistency before assignment. |
| API generates exercises without pedagogy | Require complete capsule structured output. |
| Feedback is generic or unhelpful | Validate brief/full feedback and supervise quality through use evidence. |
| Step-by-step is missing or shallow | Require step-by-step in the same output and validate required parts. |
| Cost scales linearly with students | Reuse living bank capsules and generate only for gaps. |
| Mauricio becomes the generation bottleneck | Use governed external API infrastructure, not personal ChatGPT workflow. |
| Generated capsules create pseudo-adaptive claims | Enforce prohibited claims and BPCPv1 language limits. |
| Bank accumulates low-quality capsules | Version, audit, promote, revise, or retire capsules based on evidence. |
| Human review blocks continuity | Use automatic validation for routine assignment and human supervision for exceptions. |
| Continuous improvement is mistaken for autonomous AI | State clearly that improvement is governed, supervised, versioned, and not adaptive AI. |
