**Date:** 2026-04-30

**Scope:** Understand POST /api/learning-cycles behavior, SkillState heuristic, auto-load generation, CycleSnapshot, enrollment.currentCycleId update — before creating Cycle 2  
**Mutation:** None

## 1. Executive Summary

Mauricio Beta-M1 / PAES_M1 / Cycle 1 is **closed** and **continuity has been authorized** (ContinuitySignal with `signalType='continue'`, `rationale='admin_authorize'` exists). The platform is structurally ready for the creation of Cycle 2.

However, Cycle 2 has NOT been created. The next operation (POST /api/learning-cycles) carries **medium risk** because it is a multi-mutation transaction that:

1. Creates a new `LearningCycle` (Cycle 2).
2. Reads `SkillState` data and applies a deterministic heuristic to select up to 3 skills.
3. Auto-generates 1–3 `StudyLoad` records with titles derived from skill names.
4. Creates a `CycleSnapshot` (snapshotType: `cycle_open`) capturing the heuristic trace.
5. Updates `enrollment.currentCycleId` and `enrollment.lastActivityAt`.

The purpose of this audit is to understand each of these side effects, evaluate whether the auto-generated StudyLoads are suitable for MVP, and determine the safest path to create Cycle 2.

**This audit does not create Cycle 2, does not create StudyLoads, does not mutate data.**

## 4. POST /api/learning-cycles Endpoint Audit

### 4.1 Path and Method
- **Path:** `POST /api/learning-cycles`
- **File:** `app/api/learning-cycles/route.ts`

### 4.2 Required Request Body
```json
{ "enrollmentId": "<string>" }
```
`status` is accepted but defaults to `"open"`. `cycleNumber` is ignored (auto-calculated).

### 4.3 Auth / Session Requirements
- Requires `getServerSession(authOptions)` → returns 401 if no session.
- No role check (any authenticated user can call this endpoint).

### 4.4 Preconditions (P1–P4b)

| Check | Code | Guard | HTTP on fail |
|-------|------|-------|--------------|
| **P1: Enrollment exists and is active** | `findUnique` + `status !== 'active'` | 404 or 400 | 404 / 400 |
| **P2: currentContinuityState is "normal"** | `enrollment.currentContinuityState !== 'normal'` | 400 | 400 |
| **P3: No open cycle for this enrollment** | `findFirst({ where: { enrollmentId, status: 'open' } })` | 409 | 409 |
| **P4a: (First cycle only) Completed initial diagnostic** | `findFirst({ where: { enrollmentId, diagnosticType: 'initial', status: 'completed' } })` | 400 | 400 |
| **P4b: (Subsequent cycles) Previous cycle closed + ContinuitySignal** | `lastCycle.status !== 'closed'` + `findFirst({ where: { enrollmentId, learningCycleId: lastCycle.id, signalType: 'continue' } })` | 400 | 400 |

**For Mauricio (Cycle 2):**
- P1: ✅ Enrollment is active.
- P2: ✅ `currentContinuityState` is `"normal"` (has not been changed).
- P3: ✅ No open cycle (Cycle 1 is closed).
- P4a: N/A (not first cycle).
- P4b: ✅ Cycle 1 is closed AND ContinuitySignal exists.

### 4.5 Transaction Body

All mutations are inside a single `prisma.$transaction`:

1. **Create LearningCycle**
   - `cycleNumber`: `lastCycle.cycleNumber + 1` → 2
   - `status`: `"open"`
   - `enrollmentId`: Mauricio's enrollment

2. **Read SkillStates for heuristic**
   - `findMany({ where: { enrollmentId, skill: { programId: enrollment.programId } } })`
   - Scoped to the enrollment's program.
   - Includes `skill: { select: { id, code, name } }`.

3. **DU: Prior-cycle skill exclusion**
   - For `nextCycleNumber > 1`, reads Cycle 1's `cycle_open` snapshot.
   - Extracts `selectedSkillIds` from the snapshot payload.
   - Filters candidates to exclude those skill IDs.
   - If exclusion leaves 0 candidates, relaxes back to the full sorted list.

4. **DN: Deterministic heuristic sort**
   - Sort by: `needsReinforcement` DESC (true first), `masteryLevel` ASC, `skillId` ASC.
   - Take top 3 (`MAX_FIRST_LOADS = 3`).

5. **Create CycleSnapshot (cycle_open)**
   - `snapshotType: 'cycle_open'`
   - Payload includes: `heuristic`, `maxFirstLoads`, all `skillStates`, `selectedSkillIds`, `excludedSkillIds`, `exclusionRule`, `exclusionRelaxed`.

6. **Create StudyLoads**
   - If selected.length > 0: creates one StudyLoad per selected skill:
     - `loadType: 'practice'`
     - `title: 'Practice: <skill.name>'`
     - `status: 'pending'`
   - If selected.length === 0: creates one fallback load:
     - `loadType: 'practice'`
     - `title: 'Initial practice'`
     - `status: 'pending'`

7. **Update enrollment**
   - `currentCycleId: newCycle.id`
   - `lastActivityAt: new Date()`

### 4.6 Response Shape
- Returns the created LearningCycle with enrollment select and `_count` of cycleDecisions, studyLoads, cycleEvaluations.
- HTTP 201 on success.

### 4.7 Failure Modes
- 401: No session.
- 404: Enrollment not found.
- 400: Enrollment not active, continuityState not normal, no diagnostic (P4a), previous cycle not closed (P4b), no continue signal (P4b).
- 409: Open cycle already exists, duplicate cycleNumber (P2002).
- 500: Internal error.

## 5. SkillState Heuristic Audit

### 5.1 Fields Used
- `needsReinforcement` (Boolean, default false)
- `masteryLevel` (String, default `"not_evaluated"`)
- `skillId` (String, tiebreaker)

### 5.2 Sort Logic
```
needsReinforcement: true first (DESC)
masteryLevel: ASC (lexicographic)
skillId: ASC (lexicographic, tiebreaker)
```
Take top 3.

### 5.3 Determinism
- **Yes, fully deterministic.** No randomness, no timestamps, no external inputs.
- Given the same SkillState data, the same selection will always be produced.

### 5.4 Masterly Level Ordering Risk
- The sort uses **lexicographic ASC** on `masteryLevel`.
- If values are `"not_evaluated"`, `"beginner"`, `"intermediate"`, `"advanced"`, the lexicographic order is: `advanced` < `beginner` < `intermediate` < `not_evaluated`.
- This means `advanced` sorts first (lowest ASC), then `beginner`, then `intermediate`, then `not_evaluated`.
- **Risk:** This is counterintuitive. "Advanced" skills would be prioritized over "not_evaluated" skills.
- **Mitigation for MVP:** If all SkillStates have `masteryLevel: 'not_evaluated'` (seed/default), the sort falls through to `skillId` tiebreaker, which is safe.
- **Mitigation for Mauricio:** Depends on actual SkillState data. If SkillStates were initialized by diagnostic completion (DM), they may have varied values.

### 5.5 Sparse/Seed-Level SkillState
- If SkillState data is sparse (few records) or all at default values (`not_evaluated`, `needsReinforcement: false`):
  - All records are equal on the first two sort keys.
  - Selection falls to `skillId` ASC, which is CUID-based and effectively arbitrary but deterministic.
  - Top 3 by skillId will be selected.
- If **no** SkillState records exist for the enrollment:
  - `selected` is empty.
  - Fallback: one `"Initial practice"` load is created.

### 5.6 DU Prior-Cycle Exclusion
- For Cycle 2, the heuristic reads Cycle 1's `cycle_open` snapshot.
- Extracts `selectedSkillIds` and filters them out.
- If Cycle 1 had 3 selected skills and the enrollment has exactly 3 SkillStates, exclusion leaves 0 → relaxation activates → the same 3 skills are re-selected.
- If the enrollment has >3 SkillStates, exclusion works as intended: Cycle 2 gets different skills.
- **Relaxation is auditable** via `exclusionRelaxed: true` in the snapshot.

### 5.7 Content Registry Match
- Generated StudyLoad titles follow the pattern: `"Practice: <skill.name>"`.
- The content registry (`lib/study-load-content.ts`) is keyed by exact title match.
- Current registry entries:
  - `"PAES M1 — Ecuaciones lineales básicas"`
  - `"PAES M1 — Problemas con ecuaciones lineales"`
- **Auto-generated titles will NOT match the content registry** because they follow the pattern `"Practice: Resolver problemas de ecuaciones lineales"` (using skill.name, not the registry key).
- **Impact:** The `/now` page's `getStudyLoadContent(load.title)` will return `undefined`. The "Ver actividad" link will NOT appear. The student will see the load card with "Empezar" and "Terminar" buttons but no interactive MC activity.
- **Risk:** Medium. The student can still start/complete the load and leave a self-report, but there is no in-app content to practice with. This is the same as Cycle 1's initial state before FL-UX content was manually created.

### 5.8 Safety for MVP Beta
- The heuristic is deterministic and auditable.
- The fallback ensures at least one load is always created.
- The masteryLevel lexicographic sort is a known limitation but acceptable when all values are `"not_evaluated"`.
- The content registry mismatch is the primary practical risk.

## 11. Product Interpretation

Cycle 2 in the MVP context should mean:

1. **Not adaptive automation.** The platform does not automatically decide what the student should study next based on performance metrics.
2. **Not score-based progression.** There is no PAES score calculation or threshold-based advancement.
3. **Manual continuation from evidence-backed Cycle 1.** The admin reviewed evidence, made a decision, closed the cycle, and authorized continuity. Cycle 2 is the next unit of regulated learning.
4. **Should probably start with one carefully controlled load.** Given that auto-generated loads lack content registry match, the ideal MVP approach is to ensure the student has a load with actual in-app content to work with.
5. **Avoid hidden auto-generation unless understood.** The admin should be aware that creating a cycle auto-generates 1–3 StudyLoads. If these loads don't have content, they are structurally valid but pedagogically empty.
6. **Preserve admin oversight.** The admin should be able to:
   - Review auto-generated loads after cycle creation.
   - Edit, delete, or replace loads as needed.
   - Create new loads with content-registry-matching titles.

## 15. Risks

### 15.1 Auto-Generated StudyLoads Without Content Registry Match
- **Severity:** Medium.
- **Impact:** Student sees loads without "Ver actividad" link. No in-app MC activity.
- **Mitigation:** Admin curates loads post-creation (Option C).

### 15.2 Masterly Level Lexicographic Sort Anomaly
- **Severity:** Low (for MVP with seed-level data).
- **Impact:** If SkillStates have varied masteryLevel values, "advanced" skills sort before "not_evaluated".
- **Mitigation:** Monitor selected skills in CycleSnapshot payload. All default/seed values are `"not_evaluated"`, making this a non-issue for Mauricio.

### 15.3 DU Exclusion Relaxation
- **Severity:** Low.
- **Impact:** If Cycle 1's selectedSkillIds cover all available SkillStates, exclusion relaxes and the same skills are re-selected.
- **Mitigation:** Auditable via `exclusionRelaxed: true` in snapshot. Admin can curate.

### 15.4 currentCycleId Update
- **Severity:** Low.
- **Impact:** If transaction fails, currentCycleId remains pointing to Cycle 1 (closed). /now shows "no open cycle" — safe.
- **Mitigation:** Transaction atomicity.

### 15.5 Duplicate Cycle 2
- **Severity:** Low.
- **Impact:** Unique constraint `[enrollmentId, cycleNumber]` prevents duplicates. Second attempt returns 409.
- **Mitigation:** Built-in guard.

### 15.6 Duplicate StudyLoads
- **Severity:** Low.
- **Impact:** StudyLoads have no unique constraint on title. If the transaction runs twice (shouldn't due to P3 open-cycle guard), loads could duplicate.
- **Mitigation:** P3 guard + transaction atomicity.

### 15.7 Loss of Visibility of Cycle 1
- **Severity:** None.
- **Impact:** Cycle 1 remains in database, accessible via admin UI.
- **Mitigation:** FK relations preserve all evidence.

### 15.8 Student Sees Empty /now or Content Unavailable
- **Severity:** Medium.
- **Impact:** Student sees pending loads without interactive content.
- **Mitigation:** Admin curation (Option C).

### 15.9 Continuity Mistaken for Adaptive Recommendation
- **Severity:** Low (product messaging risk).
- **Impact:** Auto-generated loads might be interpreted as "the system recommends these skills."
- **Mitigation:** Clear documentation that this is heuristic-based, not adaptive. Admin curation overrides.

### 15.10 Schema / Deploy / Data Mutation Risks
- **Severity:** None for this phase (audit only).
- **For FL-UX-3E-OP:** The Cycle 2 creation is a data mutation via existing API. No schema change, no deploy needed.

### 15.11 Abacus Auto Schema Promotion Risk
- **Severity:** None. No schema changes in this phase or the recommended next phase.

### 15.12 Checkpoint Artifacts
- **Severity:** None. Documentation-only changes. No build artifacts affected.
