## 15. Rationale and distractor-rationale boundary

Future first-batch authoring may write final documentation-level rationales for capsules 2, 3, and 5.

Each item should receive:

- item-level rationale;
- rationale for why the candidate key works;
- distractor-level rationale, or an explicit error-family rationale when distractor-level detail is not stable enough;
- inference limit.

These rationales must remain documentation artifacts until implementation is explicitly authorized.

They must not:

- diagnose the student;
- claim what the student knows or does not know;
- create mastery/readiness language;
- change item content silently;
- imply registry readiness.

## 16. Feedback breve authoring boundary

Future first-batch authoring may write final documentation-level feedback breve for each item in capsules 2, 3, and 5.

Feedback breve must:

- acknowledge result without overclaiming;
- name the task focus;
- give one concise central idea;
- give one concrete next action;
- use cautious evidence language;
- be short enough for mobile-first use.

Feedback breve must not:

- be only "correct/incorrect";
- diagnose;
- claim mastery/readiness/theta/score;
- decide route;
- say Bexauri knows the student's level.

## 17. Feedback completo authoring boundary

Future first-batch authoring may write final documentation-level feedback completo for each item in capsules 2, 3, and 5.

Feedback completo must include:

- what the item asked;
- why the correct option works;
- why the selected wrong option may be tempting, if applicable;
- central concept or relation;
- cautious interpretation of the error;
- short review recommendation;
- inference limit.

Feedback completo must not:

- become a long generic lecture;
- imply a diagnosis;
- claim mastery/readiness/theta/score;
- change the item stem or options;
- create student-facing route certainty.

## 18. Step-by-step authoring boundary

Future first-batch authoring may write final documentation-level step-by-step scripts for capsules 2, 3, and 5.

Each step-by-step script must:

- be item-specific;
- have a maximum of 4-6 steps;
- distinguish calculation from interpretation;
- end with a check;
- be mobile-readable;
- preserve cautious evidence language.

Step-by-step scripts must not:

- expose final implementation keys as implementation-ready assets;
- become generic lectures;
- replace feedback completo;
- imply adaptive AI;
- change source content.

## 19. Mobile readability acceptance checklist

Future authoring output must pass this checklist:

- Each feedback breve is short enough for a mobile card.
- Each feedback completo is segmented and scannable.
- Each step-by-step script has 4-6 compact steps.
- No paragraph is required to understand the first action.
- Mathematical notation is readable in plain text.
- Item-specific wording does not create excessive cognitive load.
- Feedback does not repeat the full item unless necessary.
- Capsule 2/3/5 authoring remains compact enough for early MVP flow.
- Any longer explanation is placed in complete feedback, not in brief feedback.

## 22. Stop conditions for future authoring

Future authoring must stop and mark a blocker if:

- an item key appears inconsistent;
- feedback requires changing item wording;
- capsule 3 or capsule 5 source-candidate status becomes blocking;
- capsule 4 or capsule 6 item count is too broad for mobile readability;
- language drifts into diagnosis, mastery, readiness, theta, score, route certainty, adaptive AI, or Sales-Ready claims;
- authoring begins to imply registry-ready implementation;
- a distractor cannot be rationalized without inventing unsupported student-state claims;
- a step-by-step script requires solving a different task than the item asks;
- mobile-readable constraints cannot be met without changing source content.
