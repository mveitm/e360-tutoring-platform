# Tutoring Platform MVP

## Project Overview
Adaptive tutoring platform MVP. Building incrementally — data model first, then UI, then adaptive logic.

## Current Phase: F — Enrollment Filters on Admin Views

### Phase E (complete): Auto-increment & closed-cycle guards

- CycleDecision, StudyLoad, CycleEvaluation POST reject if linked cycle is closed (400)

## Current Phase: D — Minimal End-to-End Operational Flow
Ten entities total, built across four phases:

### Phase A (complete): Core operational entities
- **Students** — basic student info, unique email
- **Programs** — PAES_L1, PAES_M1, PAES_M2 pre-seeded
- **StudentProgramInstance (Enrollments)** — core operational entity linking students to programs with lifecycle fields (startedAt defaults to now, endedAt/lastActivityAt/currentCycleId nullable, currentContinuityState defaults "normal")

### Phase B (complete): Pedagogical structure
- **Axes** — high-level learning axes per program (unique programId+code)
- **Skills** — specific skills per axis+program (unique programId+code)
- **SkillStates** — per-enrollment skill state tracking (unique enrollmentId+skillId). Linked to enrollments, NOT directly to students.

### Seed Data (PAES_M1 only)
- Axes: ALG (Álgebra), GEO (Geometría), DAT (Datos y probabilidad)
- Skills: ALG_LINEAR, ALG_FUNC, GEO_ANGLE, DAT_TABLE

### Phase C (complete): Adaptive cycle skeleton
- **LearningCycle** — per-enrollment cycle unit (unique enrollmentId+cycleNumber). Fields: cycleNumber, status (open/in_progress/closed), openedAt, closedAt.
- **CycleDecision** — decision registered per cycle. Fields: decisionType (advance/reinforce/hold/redirect), rationale (optional).
- **StudyLoad** — work block per cycle (multiple per cycle). Fields: loadType (practice/review/diagnostic/reinforcement), title, status (pending/in_progress/completed), releasedAt, dueAt.

### Validated chain
`StudentProgramInstance → SkillStates → LearningCycles → CycleDecisions → StudyLoads`

### Phase D (complete): Minimal end-to-end operational flow
- **CycleEvaluation** — evaluation per cycle (multiple per cycle). Fields: evaluationType (diagnostic/progress_check/cycle_close), resultSummary (optional).
- **PATCH endpoints** for inline status/state updates:
  - `PATCH /api/learning-cycles/[id]` — status update (auto-sets closedAt when closed)
  - `PATCH /api/study-loads/[id]` — status update
  - `PATCH /api/skill-states/[id]` — masteryLevel, confidenceLevel, needsReinforcement, stateSource (auto-sets lastEvaluatedAt)
- **Active cycle convention**: query-time (status=open OR in_progress for enrollment), no FK constraint on currentCycleId
- **Admin UI**: inline select dropdowns for status/mastery on existing views; new CycleEvaluation CRUD view; nav updated with Evals in Cycles group
- **Validated chain**: `Enrollment → SkillStates → LearningCycle → CycleDecision → StudyLoads → CycleEvaluation → close cycle → open next`

## Technical Decisions
- Database: PostgreSQL via Prisma ORM
- Auth: NextAuth.js with email/password credentials
- Admin UI: Minimal card-based CRUD views per entity
- Navigation: Grouped header nav (Core | Pedagogy | Cycles) — 10 sections total (Evals added in Phase D)
- All admin routes protected via middleware
- Seed script uses upsert to avoid duplicates
- No seed data for cycle entities (operational, not reference data)

## Design Language
- Minimal card-based admin interface
- Consistent pattern: list view + dialog-based create form per entity
- Lucide icons per entity type (RefreshCw for cycles, GitBranch for decisions, FileBox for loads, ClipboardCheck for evaluations)
- Badge for status display
- Toast notifications via sonner

## NOT YET BUILT (future phases)
- Sessions, responses, incident_logs, audit_events
- Adaptive logic / scoring formulas
- Student-facing UI
- Dashboards / analytics
- Chatbot flows
- Google SSO, async processes
